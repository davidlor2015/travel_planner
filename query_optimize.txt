## Query Performance Audit Summary
Audit scope covered all requested backend flows through route → service → repository/model + Alembic index coverage.  
Current state is generally solid on indexing (notably recent 2026-04-30 index migrations), with the main launch risks concentrated in query shape and over-fetching, not missing core indexes.

## Core Flow Findings

### 1. Trip Workspace Load
- Files:
  - [trips.py](/mnt/c/users/david/desktop/planner_proj/travel-planner/app/api/v1/routes/trips.py)
  - [trip_service.py](/mnt/c/users/david/desktop/planner_proj/travel-planner/app/services/trip_service.py)
  - [trip_membership_repository.py](/mnt/c/users/david/desktop/planner_proj/travel-planner/app/repositories/trip_membership_repository.py)
  - [trip_repository.py](/mnt/c/users/david/desktop/planner_proj/travel-planner/app/repositories/trip_repository.py)
- Tables:
  - `trips`, `trip_memberships`, `trip_member_states`, `trip_invites`, `itinerary_days`, `itinerary_events`, `itinerary_day_anchors`, `packing_items`, `budget_expenses`, `prep_items`, `reservations`
- Findings:
  - `GET /v1/trips/{trip_id}/member-readiness` builds planning readiness via `joinedload` of 3 member-state collections at once ([trip_membership_repository.py:64](/mnt/c/users/david/desktop/planner_proj/travel-planner/app/repositories/trip_membership_repository.py:64)).  
    Problem: row multiplication risk as members/items grow.  
    Why it matters: workspace collaboration call is user-facing and frequent.  
    Recommended fix: switch readiness planning loader to `selectinload` for `packing_items`, `budget_expenses`, `prep_items`; keep `joinedload` only for single-row relations.  
    Risk level: `P1`  
    Migration needed: `No`
  - `GET /v1/trips/` returns full `TripResponse` graph (`members`, `pending_invites`) ([trips.py:34](/mnt/c/users/david/desktop/planner_proj/travel-planner/app/api/v1/routes/trips.py:34), [trip_repository.py:49](/mnt/c/users/david/desktop/planner_proj/travel-planner/app/repositories/trip_repository.py:49)), while mobile list flow calls this endpoint directly ([api.ts:27](/mnt/c/users/david/desktop/planner_proj/travel-planner/ui-mobile/features/trips/api.ts:27)).  
    Problem: avoidable over-fetch for list/switcher use.  
    Why it matters: higher DB + serialization + payload cost on every trip list load.  
    Recommended fix: move list clients to `/v1/trips/lite` or slim default list response.  
    Risk level: `P1`  
    Migration needed: `No`
  - `GET /v1/trips/summaries` uses correlated subqueries and `func.date(start_at)` ([trip_repository.py:96](/mnt/c/users/david/desktop/planner_proj/travel-planner/app/repositories/trip_repository.py:96)).  
    Problem: date-cast can reduce index efficiency for upcoming-reservation metric.  
    Why it matters: called in main workspace/trips flow.  
    Recommended fix: compare `start_at` against a timestamp boundary instead of `func.date(start_at)`.  
    Risk level: `P2`  
    Migration needed: `No`
- Recommended fixes:
  - Prioritize readiness loader shape and trips list payload reduction.
  - Keep summary query single-roundtrip, but remove date-cast expression.

## Index Review
- Covered well for requested lookup patterns:
  - `trip_id`: memberships, itinerary_days, reservations, invites, execution events.
  - `user_id`: trip_memberships (`user_id` + unique `trip_id,user_id`), execution `created_by_user_id`.
  - `day_id`: itinerary_events/day_anchors.
  - `stop_ref`: execution events + composite `(trip_id,kind,stop_ref,id)`.
  - reservation trip FK + sort composite: `(trip_id,start_at,created_at)`.
  - packing/budget/prep now scoped by `member_state_id` with indexes.
  - invite token and pending lookups: unique token + email/status/trip composites.
- Potential optional gap:
  - `trips.user_id` is not indexed in model/migrations ([trip.py:19](/mnt/c/users/david/desktop/planner_proj/travel-planner/app/models/trip.py:19)); not a clear core-flow bottleneck today, so I would not add yet.

## N+1 Query Risks
- `GET /v1/trips/{trip_id}/on-trip-snapshot`: membership user access in loop may issue per-member lazy queries.
- `GET /v1/trips/{trip_id}/member-readiness`: not classic N+1, but high row-amplification risk due multi-collection `joinedload` in one query.

## Endpoints Loading Too Much Data
- `GET /v1/trips/` returns full trip graph for list/switcher consumers that do not need it.
- `GET /v1/trips/{trip_id}/member-readiness` loads full planning collections for every member, every call.
- `GET /v1/trips/{trip_id}/on-trip-snapshot` does duplicate member-email retrieval work.

## Recommended Fix Priority

### P0
- None identified in current code.

### P1
- Optimize on-trip snapshot member loading/reuse to remove poll-time N+1 risk.
- Change member-readiness loader strategy from multi-collection `joinedload` to `selectinload`.
- Reduce `/v1/trips/` over-fetch in list contexts (`/lite` usage or slim list response).

### P2
- Improve summaries upcoming-reservation filter to be index-friendly (remove `func.date(...)`).
- Keep full-replace itinerary apply for launch, but instrument latency and payload size.
- Use lightweight auth-only membership checks on hot write paths.
- Make packing suggestions query read only reservation types needed.

## Suggested Implementation Plan
1. Fix snapshot query shape (`on-trip-snapshot`) first because it is polling-driven and user-visible.
2. Refactor member-readiness loading to avoid row explosion; verify identical response contract.
3. Switch list consumers to `/v1/trips/lite` (or add a new slim default list path) without changing detail API.
4. Clean up summary query date predicate for index friendliness.
5. Add telemetry tests/benchmarks for snapshot/readiness/list endpoints before and after changes.
